which one are we doing?

The biggest product gap in the live app → our designs. Lodging is the one line item where voting matters, because lodging is the thing the crew actually argues about. Left: the voting-open state (pre-lock). Right: the winner-locked state (post-lock). Losers stay visible, crossed out, so everyone can see what was on the table. Everything else in the trip page is unchanged — this spec only defines the lodging card.

Smart-link images (critical)

Each option pulls its hero image via the existing smart-link system. The image is the single biggest UX lever on the voting card — people vote with their eyes. The placeholder gradient in these specs stands in for the real Airbnb / Vrbo / Booking image pulled at link-paste time.

Rules:

  • 16:9 aspect ratio, full-bleed top of the option card, hard 2px border below (chassis stroke).
  • Source-domain pill bottom-left of the image (e.g. "airbnb.com", "vrbo.com"). Tiny trust signal.
  • If the smart-link fetch fails, fall back to a generic colored chassis panel with a Caveat label of the domain. Never show a broken-image icon.
  • Loser state: grayscale + 0.7 brightness + a 0.25 black overlay. The image is still readable — just clearly past-tense.
  • Winner state: image stays full-color, full brightness. The 🗝️ lock-stamp overlay sits in the body, not on the image.
  • The image IS the link. Tap anywhere on the image → opens the original listing URL in the system browser (new tab). Visible affordances on the image: source-domain pill with ↗ arrow (bottom-left), "see it →" Caveat pill (bottom-right). Both are decorative — the entire image is the hit target. No in-app browser for v0.
  • The option title is also tappable and routes to the same URL. Price and meta are not.

Mechanic

Who can add options: organizer only (v0). Future: allow any crew to suggest.

Who can vote: any crew member who has RSVP'd "i'm in" or "hold my seat". Out = no vote.

One vote per person. Changing vote moves it; no multi-select.

Resolution: organizer taps "lock it in" on any option. Winner gets 🗝️ + lock stamp, losers cross out but stay visible.

After lock: card becomes read-only. Losing options are archived-in-place, not deleted, so the story of the decision is visible.

Min options to enter voting mode: 2. A single lodging item is just a regular line item, no vote UI.

claude code handoff notes

Data model: lodging line item has an options array (0, 1, or 2+). If 0–1, render as a normal line item. If 2+, render as voting card.

Vote state: { option_id: voter_ids[] }. One vote per user across the whole card (not per option).

Lock state: locked_option_id + locked_by + locked_at. Presence of locked_option_id switches the card to read-only.

Tally bar percent: option's votes ÷ total votes cast (not ÷ total crew). A close race shows 50/50, not 50/25 when half the crew hasn't voted.

Who can lock: organizer only. Button lives inline on each option, only visible to organizer, surfaces on tap of the option.

Edge case — no votes: organizer can still lock any option. Tally bars render as 0%.

Edge case — tie: no auto-resolution. Organizer picks. Ties don't need a special UI.

Nudge hook: if voting has been open for >72h and <50% of crew has voted, nudge fires: "3 people still haven't weighed in on where we sleep".
