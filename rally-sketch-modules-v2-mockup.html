design notes — what's canonical here

module order (top → bottom)

  1. marquee strip
  2. trip hero (title · dates · crew count)
  3. getting here (helper text only at sketch — 8I)
  4. the headliner (optional, singular trip-level — 8J)
  5. the spot (lodging)
  6. transportation (rebuilt with flight-as-type + multi-leg — 8I)
  7. activities (single per-person estimate, matches provisions — 8K)
  8. provisions (single per-person estimate)
  9. cost summary

getting here (8I)

  • Helper text only at sketch — no input, no card, no drawer, no cost contribution.
  • Copy: "each crew will pull their own arrival estimate in sell — however they're getting here."
  • Sell-phase arrival estimator (per-crew, Google Flights deep-link for flyers, different prompts for drivers/train/etc.) is deferred to Session 9+.

transportation (8I)

  • One module. Flight is a type_tag, NOT its own module.
  • Each line: description, cost_cents, cost_unit (per_person | total), type_tag (optional), link_url (optional, plain text — no enrichment).
  • Type tags (7): flight · rental · train · van-bus · ferry · charter · other.
  • Default cost_unit: per_person for flight & train; total for rental / charter / van-bus / ferry / other. Organizer can override.
  • Multi-leg handled natively: multiple line items, one per leg.
  • flights table is NOT dropped — retained for sell-phase work.

the headliner (8J)

  • Optional, singular trip-level component (columns on trips, not a related table).
  • URL paste → /api/enrich → OG image + title auto-fill. Reuse lodging pattern, don't fork.
  • Cost unit toggle (per_person / total) stored on the record.
  • Renders FIRST in cost summary breakdown with visual emphasis.
  • Theme-agnostic component — accent color only variation, no per-theme copy / icons.

activities (8K)

  • Single per-person estimate field. No line items, no drawer, no cards.
  • Matches the provisions module shape exactly.
  • Column: activities_estimate_per_person_cents on trips.
  • activities table is NOT dropped — retained for sell/lock activity mechanic later.

cost summary math

  • Per line: per_person → adds cost_cents directly. total → adds cost_cents / in_crew_count.
  • Headliner line renders first. Transportation lines list individually (not aggregated). Activities + provisions as single per-person figures.
  • All math in one helper (lib/cost-summary.ts). No duplication across components.
  • Caveat line in the summary: "excludes arrival — each crew estimates in sell."

multi-leg edge case — capability, not primary shape

The Coachella populated frame above represents the core single- destination case (~14 of 17 themes). The transportation module also supports multi-leg trips (Euro Summer shape) via multiple line items with mixed cost units — but this is an edge case the module flexes to handle, not the primary reference shape. Do NOT design the module around multi-leg; design it around the core case and verify it still handles multi-leg as a consequence.

Example edge-case transport list (for reference — do NOT mock this in the UI):

  • flight · rome → barcelona · $120 / person
  • train · barcelona → paris (tgv) · $90 / person
  • rental · tuscany week · $480 / total

Each line contributes to the per-person summary independently based on its unit. Multi-leg lodging ("the spot" across 3 cities) is NOT supported — see multi-leg deferral note in rally-fix-plan-v1.md.

what's OUT of scope for 8I / 8J / 8K

  • Sell-phase per-crew arrival estimator (Session 9+).
  • Google Flights deep-link or any flight pricing API/integration.
  • Multi-leg lodging (the spot stays singular — see multi-leg deferral note in fix plan).
  • Dates / assignees / "who's on this" on transportation lines.
  • Per-line cost-split UI beyond the per_person/total toggle.
  • URL enrichment on transportation (confirmation pages don't have useful OG data).
