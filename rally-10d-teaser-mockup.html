Call 2 — post-tap state mechanics

What happens on tap:

  1. Button enters loading state (Shrikhand text dims, brief pulse) for ~300ms while the magic-link request fires.
  2. Server calls signInWithOtp({ email: trip_members.email }) using the email already known from the resolved token. No email input UI — the invitee's identity is pre-known.
  3. On success, sticky bar morphs from "see the plan →" CTA to the "✓ link sent to j**@example.com" status pill (with masked email). Resend link below shows 30s cooldown.
  4. Locked-overlay copy updates from "sign in to see the plan ↑" to "check your email ✉" with the lime accent treatment to signal "next step is in your inbox" without changing the lock metaphor.
  5. Email entry rest of teaser unchanged. URL stays at /i/<token>.
  6. Client subscribes to Supabase's auth state via onAuthStateChange. When the magic-link click in the inbox completes the auth round-trip (in this tab OR another tab via storage events), the listener fires → unblur reveal animation begins → page transitions to full sell view inline.

Error state: if signInWithOtp fails, the sticky bar shows "couldn't send · try again" instead of the success status, and tapping the bar re-fires the request.

Cooldown: resend link is disabled for 30s after each send (mirrors phase 11 state 2's existing behavior). Same lexicon, same timer.

Edge cases (v1): stale email (user lost access), wanting a different email, magic link expired before click — all flagged for v1 and not handled in 10D.

Call 1 — locked: hybrid blur + overlay

Hybrid blur + translucent overlay (option (c)). Polished treatment ports verbatim from phase 5. CSS specifics (filter: blur(4px), border: 2.5px solid var(--ink), box-shadow: 3px 3px 0 var(--ink), transform: rotate(-2deg) on the lock-msg) all stay authoritative.

Call 3 — locked: unblur reveal animation

Total reveal duration: 600-800ms.

Sequence (parallel + offset):

  1. Blur drop: filter: blur(4px) → blur(0) over ~600ms ease-out.
  2. Lock pill + overlay message fade: starts ~200ms before blur completes, ~200ms duration. Pill scales down (1 → 0.92) while opacity goes 1 → 0. Out before the rows are sharp so the user sees content emerge cleanly.
  3. Sticky bar transition: "✓ link sent" status pill cross-fades to the existing StickyRsvpBarChassis three-button RSVP bar (in / holding / out). ~200ms cross-fade. Triggered when the auth listener fires; runs concurrent with the blur drop.
  4. Page-level transition: after the unblur completes, router.replace('/trip/<slug>') silently updates the URL. No loading flash because the teaser DOM IS the trip view at that point — the redirect is a URL-only update for back-button + share-link consistency.

Reduced-motion accessibility: respect prefers-reduced-motion: reduce by skipping the animation entirely. Instant snap from blurred → unblurred when auth completes. Industry-standard a11y default.

Call 4 — locked: blur cutline + going row removed

Sharp (above the blur): marquee, sticker, inviter chip, wordmark, eyebrow, title, tagline, countdown card. Establishes trip vibe + identity + dates + organizer. Answers the questions: where? when? from whom?

Removed entirely from the teaser: the going row (avatars + "5 already in / 1 seat with your name"). Phase 5 used this as social proof; 10D removes it to keep RSVP context mysterious — pure trip-vibe FOMO, no influence from "who else is going." Different conversion bet.

Blurred (the lock): the entire .locked-section — flight + lodging + activities + your share total. The thing the invitee has to commit to financially is the thing they sign up to see.
