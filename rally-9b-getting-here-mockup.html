Why I kept the gh-* prefix on the three new primitives. They are Getting-Here-specific shapes with no existing precedent (mode-picker grid, passport-origin nudge, personal roll line). Using a namespaced prefix marks them as "new to this module" rather than promoting them to global primitives prematurely. If a later module reuses one (e.g. transport grows its own mode picker), we can generalize then.

Required-soft enforcement

Per Andrew's call: arrival must be filled in, but not as a hard gate. Behavior:

  • Empty state (mode = null): .module-section-empty + mode picker visible. No input yet.
  • In-progress (mode set, cost null): picker active + dashed .estimate-input + roll line reads "(pending)".
  • Complete: solid .estimate-input.filled + roll line shows amount. 9B-2 will also flip the cost summary's "your way in" row from pending → real number.
  • No RSVP button disable. No red-outline blocker. Friction is visual only.
Escalation triggers for the brief. (a) If the target table for migration isn't trip_members (could be members, trip_invitees, etc. — schema-dependent), flag before running the migration. (b) If passport.based_in isn't already queryable from the trip page's existing query (8D shipped it, but join shape matters), flag — may need a query extension. (c) If the reference-link URL templates fail-open on odd destinations ("palm spring, ca" — spaces, no country), Google still opens but link quality may vary. Accept URL-encode + hope. Not a blocker. (d) If mode-picker tiles feel too cramped at 375px, flag — may need 2×2 grid instead of 1×4. Keep current design as preferred. (e) If "other" mode with cost=0 feels visually dead (no link, no hint), consider a minor variant. DO NOT add a note field in 9B-1.
