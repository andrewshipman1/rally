design notes — drawer behaviors CC needs to get right

field order (unusual)

  • Link is first, not description. Intentional: most users paste a URL and let enrichment fill the rest.
  • Description is editable even after enrichment — auto-fill is a starting point, not a commitment.

enrichment flow

  • Fires on paste OR blur of the link field. Debounce ~250ms on paste.
  • Non-blocking: form stays usable during the fetch. User can type in the cost field while enrichment is pending.
  • Shimmer placeholder for the image while loading. Inline "pulling details…" text under the link field.
  • On failure: silent. Description stays empty, user types manually. No error toast.
  • Reuses the lodging enrichment pattern — /api/enrich endpoint, same response shape. DO NOT rebuild.

cost unit toggle

  • Segmented control — two options: / person and / total.
  • Default: / person (matches the common cases — tickets, tee times, retreat passes are almost always per-person priced).
  • Stored as headliner_cost_unit enum on the trips row.
  • Affects cost summary math: per_person adds directly, total divides by in_crew_count.

save / update / remove

  • Primary CTA copy: "save the headliner" (add) / "update" (edit).
  • Save disabled until description + cost are both present.
  • Remove in edit mode only. Destructive — requires a second tap to confirm (confirm bar appears below the remove button).
  • Remove clears all 6 headliner columns — does NOT delete the trip.

what this wireframe is NOT showing

  • Drawer mechanics (drag handle behavior, backdrop dismiss, Esc key, body scroll lock) — those are 8F primitives, already built.
  • Component rendering on the trip page — see rally-sketch-modules-v2-mockup.html.
  • Per-theme accent color variations — same drawer shape across every theme, only accent color changes.
