The four views, narratively

1. Logged-out teaser (the hook)

Invitee opens the share link. Sees: trip name, who called them up, dates, destination, theme postcard, countdown. Crew row visible — shows social proof ("3 already in"). Everything else (cost, lodging options, modules, buzz) is blurred. Primary CTA: see the plan →. Tap → passwordless signup.

The blur scope toggle above lets you compare "conservative" (hero + crew visible, only the plan is blurred) against "aggressive" (everything below hero fogs). My rec: conservative. The crew row is the highest-converting social-proof element we have — blurring it leaves conversion on the floor.

2. Signed-in, pre-RSVP (the pitch fully loaded)

After signup, the plan unblurs in place. "You?" avatar appears empty in the crew row. Sticky bar swaps from login CTA to 3-state RSVP: i'm in / maybe / out. Full cost summary, lodging voting, modules all readable. Buzz compose hidden (compose unlocks post-RSVP in this first pass — see open questions).

3. Crew (RSVP'd in)

Confetti pulse on commit. "You?" slot becomes real avatar. Sticky bar collapses to current RSVP state with a small "change" affordance. Buzz compose unlocks. This is the committed state — lock page is where it goes from here.

4. Organizer view

No RSVP sticky bar (organizer can't RSVP to their own trip). lock it in buttons appear under each lodging option. Share-link CTA prominent. Live activity row reads "trip is live · 3 in."

Decisions locked in this revision (v2)

  • Blur scope: conservative. Crew row visible, plan blurred. Aggressive toggle kept for contrast only — not the shipping default.
  • Module order matches the sketch page. Ordered top-down: (1) the crew, (2) the headliner, (3) the spot, (4) getting here, (5) transportation, (6) everything else (activities + provisions), (7) the aux. Cost summary tallies at the very bottom. "Getting here" (individual arrival) sits between "the spot" and "transportation" — the logical grouping is: where we stay → how I get there → how we move around once there. This mirrors the shipped sketch-page convention so sell doesn't invent a new layout grammar — it just loads the sketch grammar with richer content.
  • Getting here is a first-class sell module (NEW). Personal-only: the viewer picks their mode (✈️ / 🚗 / 🚆 / other) and drops in a rough cost estimate. That "your way in" rolls into a personalized total. Reference link adapts to the selected mode (Google Flights, Maps, Amtrak, etc.) — reference/ballpark only, not a booking path. This pulls the personalized-total mechanic forward from lock → sell. Crucially, the module does NOT show a roster of everyone's arrival methods on sell — that's a lock-phase feature (see Q10). Sell is "what's my cost"; lock is "how is the crew actually arriving."
  • Cost summary: bottom of the page. Sell the fun first, tally at the end. Cost is the closer, not the opener.
  • Countdown: scoreboard tiles, live tick. Replaced the single "51 days to lock it in" line with a yellow-stickered scoreboard (d : h : m : s), Georgia italic numerals, live-ticking seconds. Creates urgency + hype without new infrastructure. Open to further theming (see Q7).
  • Sticky bar: per-view. Teaser → "see the plan." Pre-RSVP → 3-state RSVP. Crew → quiet "you're in" confirmation. Organizer → "lock it in" / "nudge." Confirmed.

Design-system usage

  • .card, .section, .headliner, .estimate-row, .cost-summary — lifted directly from rally-sketch-modules-v2-mockup.html (the 8N .module-section idiom). Same token set: --bg, --ink, --accent, --surface, --border, --sticker-bg.
  • Countdown scoreboard uses the same token set: yellow accent fill, 2.5px ink border, 12px radius, press-shadow 3px 3px 0 var(--ink). Tiles are mini .cards — no new primitives, just composition.
  • Georgia italic for the trip title, cost total, cost pill, headliner title.
  • Voting primitives (.vote-bar, .vote-btn) match patterns from rally-phase-8-lodging-voting.html.
  • Press-shadow CTAs (3px 3px 0 var(--accent)) from the 8R invite-button rebuild.
  • New primitives introduced: .blur-veil, .lock-overlay, .sticker-called-up, .confetti, .buzz-event. All sized and colored off existing tokens — no new colors.

Motion / liveness budget

Where the "feel alive" commitment is spent:

  • Live-dot pulse in the live row — subtle, always on.
  • Marquee scroll (not animated in this mockup — would animate in real build).
  • Reveal animation from view 1 → view 2: blur-out transition, ~800ms. Crew "you?" avatar slides in. Sticky bar swaps with a cross-fade.
  • RSVP confetti on view 2 → view 3: confetti burst from sticky bar, haptic feedback, avatar morphs from "you?" dashed to filled "you" with accent fill. Maybe a brief "locked and loaded" flash on the crew section count.
  • Vote bar fill animation when tapping a vote button — fill eases from current % to new %.
  • Countdown tick — scoreboard-tile seconds bump every tick, minute/hour/day roll naturally. Lock emoji wobbles on a 3s loop.

What we're NOT spending motion on: parallax scroll, page transitions, full-page hero video, section reveal-on-scroll (all cost without conversion payoff for mobile web).

Explicit non-goals shown in wireframe

  • No per-person travel-cost field. Cost summary note says "getting to the meetup isn't included yet — we'll dial that in once it's locked." That framing sets expectation that the personal total arrives in lock phase.
  • No arrival mode picker, no Google Flights deep-link, no passport integration.
  • No restaurants/groceries breakdown — provisions stays as single estimate line.
  • No buzz reactions, no @mentions, no threaded replies.
  • No +1 guest flow, no per-person notes.
  • No voting nudges (automatic), no "arrival checked: X of Y" indicator.
  • No cost-split toggle per line item.

Open questions (pinned for Session 9 scoping)

Q1 · Login mechanics. Phone OTP, email magic link, or both? Does /auth already handle passwordless inline, or does sell need a dedicated "quick signup" flow that keeps them on the trip page?
Q2 · Blur scope cutline. Conservative vs. aggressive (toggle above). My rec: conservative — keep crew row visible for social proof.
Q3 · Buzz compose for signed-in but not RSVP'd. This wireframe hides compose until RSVP. Alternative: visible-but-disabled with "rsvp to chime in" helper, or always open. Which?
Q4 · Post-signup, pre-RSVP return visits. If someone signs in but bounces without RSVPing, what do they see on visit 2? Same pre-RSVP view? A "you're still on the fence" nudge? Decay to teaser after N days?
Q5 · Organizer's sticky bar. Nothing, a share-link CTA, or a "lock it in when ready" CTA once quorum is met? Right now I show nothing, which may be dead space on mobile.
Q6 · Invitee decline path. "Can't make it" button in the teaser sticky — do we let a logged-out invitee decline without signing in? Low-friction honesty, but no signup, so no growth-loop value. Probably yes, but flag.
Q7 · Countdown theming. Current treatment is a yellow-stickered scoreboard with ticking tiles — generic but on-brand. Alternatives worth exploring: (a) theme-adaptive numerals (e.g. pine-tree ornaments for a ski trip, beach-ball motif for tropical); (b) flip-clock analog style; (c) postcard-date-stamp visual with a "ticking stamp" effect. Depth of custom work here scales with ambition — flag how far we want to push it.
Q8 · Buzz-feed placement. Sketch page has six canonical modules (crew, headliner, spot, transport, everything else, aux). Buzz is a sell-only liveness surface with no sketch-page parent. Currently placed between "everything else" and "the aux" on the wireframe. Alternatives: (a) above the cost summary as a final "vibes" ribbon; (b) between "the crew" and "the headliner" as social proof; (c) collapsed into the crew section with a "recent activity" tab. Current placement is a guess — decide before Session 13 brief.
Q9 · Theme postcard slot. The gradient block under the dates is a placeholder for a "theme postcard" — the destination mood visual. Options: (a) on-brand abstract/gradient keyed to theme color (current), (b) curated destination photo via Unsplash keyword search, (c) AI-generated theme image per trip, (d) map snippet of the destination, (e) kill the slot entirely and let the hero breathe. v0 recommendation: keep (a), upgrade to (b) in lock phase.
Q10 · Lock-phase rescope (follow-up from Getting Here). Moving "your way in" to sell rewrites what lock was going to do. Sell now owns: personalized total, mode picker, travel-cost input, mode-specific reference links. Lock's remaining scope:
  • Full crew arrival roster. Once locked, show every attendee's method + cost in a read-only list ("andrew · ✈️ flight · $420," "robert · 🚗 drive · $85," "jane · 🚆 train · $140"). This is the "how's everyone getting there" social view — the list I originally built on sell belongs here instead.
  • Flight times + arrival details. Once each attendee confirms their booking, capture flight numbers / arrival times / train schedules / ETAs. The roster extends to show "lands at 2:14pm," "arriving wednesday 6pm," etc. Good for meetup coordination.
  • Firming estimates from "estimated" → "confirmed" across the board.
  • Final trip details — meetup addresses, access codes, itinerary polish.
  • Payment collection if/when added.
  • The lock-in commit moment itself (organizer action).
The "Sell Page Product Strategy" block in the fix plan reflects this. A dedicated lock-phase direction block will be written once sell ships.
Q11 · RSVP-before-cost-input ordering. Can someone RSVP "in" before filling out their getting-here row, or is the row required to commit? Two paths: (a) optional at sell — RSVP first, fill in "your way in" whenever you want, your total shows as "— until you set it"; (b) soft-required — sticky bar shows "rsvp" as disabled until you've picked a mode + entered a number, with copy "set your way in to rsvp." Path (a) is lower friction, (b) forces the commit to be informed. Path (a) probably wins — the whole curiosity-gap argument says we don't want friction at the RSVP tap — but worth calling out explicitly.

Proposed session breakdown (rough, pre-scope)

Each lettered item is a candidate Session 9+ brief. Ordered roughly by conversion impact per dev cost:

  1. Session 9 · invitee first-touch. Ship the logged-out teaser → signup → reveal flow. Blur veil, lock overlay, passwordless auth wiring, reveal animation. Delivers views 1 and 2. Biggest conversion lever.
  2. Session 10 · cost-summary sell-page hero. Promote CostBreakdown to the sell-page visual anchor (yellow total, Georgia italic, dark card). Wire it to read from every shipped module's sketch inputs. Add the "getting to meetup isn't included yet" caveat copy. Polish only — no new math.
  3. Session 11 · rsvp sticky bar + confetti polish. Richer press states, motion on tap, confetti burst, haptic (where available), crew-section "you?" → "you" transition. Delivers the commit moment.
  4. Session 12 · lodging voting tightening. Harden vote-bar animation, organizer lock flow, "3 votes" meta copy, lock stamp placement. Voting is already shipped — this is polish + edge cases.
  5. Session 13 · buzz feed liveness. Event row polish, "live" section badge, scroll-into-view on new event, basic compose (gated per Q3). No reactions, no mentions.
  6. Session 14 · marquee + countdown + sticker motion. The "page feels alive" pass — marquee animation, countdown scoreboard polish (optional theming from Q7), called-up sticker entrance, live-dot refinement. Low-cost polish that compounds.
  7. Session 15 (NEW) · getting-here per-crew arrival module. Ship the per-crew arrival approximator: mode chips (✈️ / 🚗 / 🚆 / other), rough-estimate input, Google Flights reference link, crew-visible rows with read-only display of other members' choices, personalized "your total" roll-up into cost summary. Moderate complexity (new per-crew data shape, new component, cost-summary rewrite), high conversion impact (this is what makes "your total" concrete). Probably slots in after Session 10 (cost summary) since they share the roll-up math.

Session 9 and 10 are the must-ships. 11–14 are compounding polish that can be re-ordered based on what we learn during 9/10 builds.
