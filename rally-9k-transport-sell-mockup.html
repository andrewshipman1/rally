Current TransportCard.tsx. Hero-style card (64px icon tile, stacked body, price block). Hardcoded English (Car Rental, Split, Per person). Hardcoded hex (#1a3a4a, #2d6b5a, #888, #999). Dead var(--rally-font-display) references. This shape never appeared in any 8I/8M wireframe — it's pre-8M legacy.

What carries over from sketch to sell

  • Card shape. Grid: 28px icon · 1fr body · auto link-chip. Single-line title with ellipsis overflow. Two-line meta row.
  • Section chrome. .module-section wrapper, .module-section-header with lowercase Georgia italic title, count pill on the right.
  • Type-tag emoji mapping. ✈️ flight · 🚆 train · 🚗 rental_car_van · 🚐 charter_van_bus · ⛵ charter_boat · ⛴ ferry · · other. Same mapping as builder/TransportCard.tsx:19–27.
  • Meta line format. $cost · split-type · type-label. Cost is formatMoney(estimated_total) for the line; split-type reads from cost_type; type-label reads from the lexicon surface keyed by type_tag.
  • Copy. Every string via lexicon. No JSX literals.

What drops on sell

  • No + add transportation button. Sell is read-only.
  • No drawer trigger on whole-card tap. Sketch's onEdit prop is not passed.
  • No .collapse-toggle — module is always expanded on sell.
  • No remove button (sketch drawer has this; sell doesn't expose it).

What's added on sell

  • Whole-card tap opens booking_link in a new tab (when present). Renders as <a href="…" target="_blank" rel="noopener noreferrer">. If booking_link is null, card renders as <div> with no cursor — non-interactive.
  • Link-chip (↗) shown only when booking_link is present. Same rule as sketch.

Design decisions (resolved 2026-04-22)

  • Flight card stays separate from TransportCard. Two files, near-identical JSX, by design. Reason: the trip.flights → transport[type_tag='flight'] data-model collapse is tracked as future work; when it lands, FlightCard.tsx is deleted entirely. Extracting a <CompactLineCard> primitive now would be speculative abstraction for a component that's scheduled to disappear. Duplication is managed with an intentional-duplication comment at the top of both files plus a Bug Backlog entry with trigger conditions (drift between files, 3rd consumer, data-model collapse) for revisiting.
  • Should Getting Here adopt the same compact card shape? It's sell-only and has its own branch-by-mode UI (9B-1). Out of scope for 9K; log as a possible later unification.
  • Confirmed badge dropped. Legacy FlightCard rendered a "Confirmed" badge when status='confirmed', but its semantic meaning was ambiguous (whose confirmation? shared flight line) and it created asymmetry — flights have a status field, ground transport doesn't, so the badge appeared unevenly across cards without clear rationale. Dropped in 9K; logged in Bug Backlog for potential redesign as a deliberate commitment-signal feature.
  • Organizer edit-on-sell deferred to a dedicated future session. Designated direction is a "back to sketch" view toggle (organizer sees a button on sell, tap toggles view to render the sketch UI, reusing the entire sketch edit surface). Client-side view toggle — NOT a new route, per three-screen rule. 9K is strictly read-only on sell for everyone; the organizer affordance gets its own session brief that also handles lodging/headliner/etc. in one pass.
