layout tradeoffs

Layout A (stacked, recommended). Identity text gets its full width and reads cleanly at the chosen copy length. The edit CTA is a distinct button on its own line — clear tap target, unambiguous. Bar is ~78px tall vs. the current ~50px, so the sell view gives up ~28px of scroll real estate.

Layout B (single-line tight). Matches today's bar height. Identity text has to drop to 13px and risks truncation on smaller screens / themes with wider diacritics. Yellow pill adds visual weight that competes with accent chrome elsewhere on the sell page (countdown tiles, cost breakdown badges).

My lean is A. The identity line is long enough that stacked treatment handles it more gracefully, and the extra 28px of bar height is a reasonable trade for clarity. But B is valid if you want to preserve scroll real estate or keep the bar visually consistent with attendee sticky-bar height.

edit-mode bar composition

Dark bg signals mode shift. The sticky bar's visual treatment changes from cream (sell/sketch normal) to --edit-bg (a deep maroon that doesn't clash with any existing theme accent). A small centered hint above the bar reads "you're editing · changes save automatically" — addresses the autosave-mental-model question without a modal or toast.

Three buttons: back · theme · done editing. Save-draft is removed from edit-on-sell mode — "draft" implies a pre-published state, but the trip is already live, and autosave already handles persistence. Publish becomes "done editing" with accent-yellow treatment as the primary action. Three buttons instead of four give the "done editing" pill proper visual dominance and leave room for back/theme icons to breathe.

"Back" during edit mode navigates to dashboard, not back to the sell view. That keeps Rally's nav model consistent (back always = dashboard). done editing is the intentional exit-to-sell shortcut. Open question below on whether back should stay or be removed entirely.

remaining open questions

1. Does "back" during edit mode stay, or get removed?
Keep: consistency with sketch (back always = dashboard), escape hatch if "done editing" feels committal. Remove: even cleaner 2-button bar (theme · done editing), no ambiguity about what "back" means during edit mode. Andrew's call.
2. Sketch-UI sell-phase compatibility audit.
Before the brief ships, audit SketchTripShell, SketchModules, SketchHeader, and the sketch-specific forms for any phase === 'sketch' checks or "set up your trip" empty states that would misbehave on a sell-phase trip. If any exist, they need a small fix in the 9W brief.
3. How does the client-side edit-mode flag persist across a page refresh?
Three options: (a) URL hash (/trip/slug#edit) — survives refresh, shareable, but weird if an invitee stumbles onto it; (b) query param (?edit=1) — same story; (c) React state only — lost on refresh. (c) is probably right — refresh exits edit mode, sane default. Decide in the brief.
