The principle

One spot module. One component. The fact that sell renders a different surface from sketch is a prop difference, not a component difference. This is the same pattern 9H applied to Headliner (readOnly prop) and the same pattern later sessions will apply to transport, everything-else, and beyond.

2 · Extend LodgingCard.tsx with a voting prop

  • New optional prop: 
    voting?: {
  currentUserId: string | null;
  isOrganizer: boolean;
  votingLocked: boolean;
  votes: (LodgingVote & { user: User })[];
  allLodging: LodgingWithVotes[];

}
  • Presence of voting = sell mode. Absence = sketch mode (current behavior). No mode: 'sketch' | 'sell' string enum — the presence discriminator is cleaner and type-inference-friendly.
  • When voting is present:
    • Hide type badge (.lodging-type-badge), remove button (.lodging-remove-btn), click-to- edit cursor.
    • Show .house-flag (winner / "not chosen") on the image when votingLocked + based on spot.is_selected.
    • Show .tally-line (vote count + voter names) below body.
    • Show .vote-row with the vote button (and organizer lock button if applicable), unless votingLocked.
  • Move the castLodgingVote + lockLodgingWinner server action imports + useTransition + handlers from the deleted LodgingGallery into LodgingCard.
  • The existing cost-math logic (computeNights, hotel per-night × rooms, etc.) applies in both modes — invitees on sell should see the same cost line as the organizer saw on sketch. Do not remove or gate it.
