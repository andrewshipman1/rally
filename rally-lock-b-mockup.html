PART 3 · EDIT-FIRST ROUND-TRIP

Exit to edit, come back to wizard

User taps edit first on the verification screen → confirmation dialog → edit-on-sell mode activates with a contextual sticky bar → user edits → taps resume lock → → wizard reopens fresh (state resets per the locked design).

13. Step A — "edit first" confirmation dialog organizer

Tapping edit first on screen 1 prompts a confirmation. Prevents accidental loss of wizard state if the user clicked the wrong thing. Note the wizard isn't "lost" anyway (resets on resume), but the confirmation makes it deliberate.

PART 4 · ERROR STATE

When fireLock fails server-side

The fire_lock RPC returns specific error codes (already_locked, not_organizer, concurrent_lock_attempt, validation errors). Most are unreachable in a normal happy-path flow — the wizard's pre-flight + CAS guards catch them. But concurrent_lock_attempt is reachable if two organizer sessions fire at once. Below: the most realistic error state.

16. Final review — concurrent_lock_attempt error organizer RPC error

Tapping lock it in 🔒 fires the action; the RPC returns { ok: false, error: 'concurrent_lock_attempt' } because another organizer session won the race. The error renders inline at the top of the review screen with a recovery affordance. Other RPC errors render the same shape with different copy.

State variants intentionally NOT wireframed (negligible visual delta): wizard screens with all-"each attendee books" allocations look identical to the right frame in Step 3 — just different item names. Skipping headliner-alloc screen when no headliner exists is the absence of that frame, not a new visual. Wizard with N transport items just shows screens 5a, 5b, 5c … with sequential progress bar advances; one transport screen wireframed (Step 6) is sufficient.
Implementation notes for the brief: • Drawer uses Rally's existing BottomDrawer primitive (LodgingAddForm precedent). • Wizard state lives in React component state, not URL. Drawer-dismiss → state reset (matches existing drawer pattern). • Progress bar fills proportionally to currentStep / totalSteps where totalSteps is computed dynamically (per gap 3 skip-on-the-fly). • Confirmation dialogs use a TBD primitive (might be a new component, or repurpose existing modal pattern — brief specifies). • Sticky-bar variant in edit-on-sell-from-wizard: track entry_point flag in URL or component state (brief specifies). • fire_lock call site: the review screen's primary CTA → fireLock server action (Lock-A exists) → RPC returns { ok, lockedAt } or { ok: false, error }.
