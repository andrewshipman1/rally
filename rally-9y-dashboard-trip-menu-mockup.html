single-item menu, deliberate

One action per kebab. Reduces cognitive load and matches the actual product rule: a sketch trip is delete-only, a sell+ trip is archive-only, an archived trip is unarchive-only. No menu offers a choice the system would just reject.

If we eventually add more actions (duplicate, share, leave-trip, etc.), the menu pattern accommodates them — but v1 keeps it tight.

why two-tap, not a modal

Two-tap respects the user's intent without breaking the dashboard scroll context. A modal would feel heavy for a single sketch trip with no invites — there's nothing meaningful to "review" before deleting. The danger color + copy change is the consent signal.

If two-tap feels too aggressive in CC's implementation (e.g., interferes with double-tap-zoom on iOS), fall back to an inline [delete] [cancel] row appearing below the menu. Both patterns satisfy the brief.

section visibility + behavior

Section header uses a dashed top-border + dimmed treatment so it reads as "below the line" relative to active trips. Cards in this section are visually de-emphasized (~65% opacity) but still tappable — the trip page loads normally on click.

The pill copy archived · was <phase> preserves the trip's pre-archive phase context (sketch / sell / lock / go / done) — useful when you're deciding whether to unarchive or scrolling through old trips.

5 · implementation notes (cross-ref the brief)
Three things this wireframe doesn't show but the brief covers, plus one decision point for CC.
not shown in this wireframe

1. Click-outside-to-close. Tapping anywhere outside the open menu closes it. Esc key closes too. Standard popover behavior.

2. Kebab keyboard accessibility. Menu items should be focusable; arrow keys navigate; Enter activates; Esc closes. Brief calls this out as an AC.

3. Direct URL behavior. Loading an archived trip's URL works for organizer + invitees. Trip page is unchanged — no "this is archived" banner. Dashboard is the only surface that signals archive state.

CC decision point

Confirmation pattern. The wireframe shows two-tap. The brief allows either two-tap OR an inline [delete] [cancel] row. CC picks based on what reads cleaner against the existing card markup. If two-tap creates iOS double-tap-zoom conflicts, fall back to the inline row. Either pattern satisfies the AC.
